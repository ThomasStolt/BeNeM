# PWA v0.1.1 Design Spec

**Date:** 2026-04-06
**Version:** 0.1.1 (from 0.1.0.5)
**Scope:** PWA (primary), middleware (proxy routes), iOS (test-connection fix)

## Overview

v0.1.1 upgrades the PWA from a read-only incident viewer to an actionable
incident management tool. Three features:

1. **Incident detail screen** — essentials-first layout with ACK action
2. **Swipe ACK/UnACK** — gesture-based acknowledge on the incident list
3. **Polished Settings** — PIN field, test-connection, grouped card layout

All write operations (ACK/UnACK, test-connection) route through the middleware
to prepare for future caching.

## Architecture

### Data flow

```
Incident list (read, unchanged):
  PWA → Caddy /bhnm → BHNM  (POST /api/incident_api.php, pwd=)

ACK / UnACK (new, via middleware):
  PWA → Middleware → BHNM  (POST /fw/index.php?r=restful/incident/acknowledge)
  PWA → Middleware → BHNM  (POST /fw/index.php?r=restful/incident/unacknowledge)

Test connection (new, via middleware):
  PWA → Middleware → BHNM  (POST /api/ha_status_api.php)
```

### Why middleware for writes?

ACK/UnACK are write operations. When caching lands (planned), the middleware
needs to know about ACK state changes to invalidate cached incident data.
Routing through middleware now avoids re-plumbing later.

### New middleware proxy routes

| PWA calls | Middleware forwards to BHNM | Auth param |
|---|---|---|
| `POST /api/proxy/incident/acknowledge` | `POST /fw/index.php?r=restful/incident/acknowledge` | `password=` |
| `POST /api/proxy/incident/unacknowledge` | `POST /fw/index.php?r=restful/incident/unacknowledge` | `password=` |
| `POST /api/proxy/ha-status` | `POST /api/ha_status_api.php` | `password=` |

These are thin pass-through proxies. The middleware forwards the request body
to BHNM and returns the response. No transformation, no caching — yet.

PWA sends `password`, `incident_id`, `user`, `comment`, `pin` (as applicable)
in the request body. Middleware forwards them as-is.

## Feature 1: Incident Detail Screen

### Scope

Essentials-first (approach B). Metadata + ACK action. No alarms, no state log
— those require `getincidentdetail` and are deferred to v0.1.2.

### Data source

Uses the existing `getincidents` response data that's already in the React
Query cache from the incident list. No additional API call needed.

The `Incident` type in `lib/api/types.ts` already has the fields needed:
`incidentId`, `displayId`, `deviceName`, `deviceIp`, `summary`, `severity`,
`status`, `incidentState`, `startTime`, `acknowledgedBy`.

### Layout

1. **Header bar** — back link + incident display ID (e.g. "#4829")
2. **Status banner** — severity badge + status tag + ACK/UnACK button
3. **Summary** — incident title/summary text
4. **Device info card** — device name, IP address, alert type (grid layout)
5. **Timing card** — created timestamp, duration, full incident ID
6. **ACK info card** — only visible when acknowledged. Green left border.
   Shows: acknowledged by (from `acknowledgedBy` field). ACK time and comment
   are not available in the `getincidents` response — they require
   `getincidentdetail` which is deferred to v0.1.2.

### ACK button behavior

- Active incident → blue "Acknowledge" button
- Acknowledged incident → outline "Unacknowledge" button
- Tapping sends ACK/UnACK via middleware proxy
- Shows spinner during request
- On success: updates local state, invalidates React Query incident cache
- On failure: shows inline error message

### File: `features/incidents/IncidentDetailScreen.tsx`

Replaces `IncidentDetailStub.tsx`. Reads incident from React Query cache by
`incidentId` route param. If not in cache (direct URL access), fetches via
`getincidents` and finds the matching incident.

## Feature 2: Swipe ACK/UnACK

### Gesture library

`react-swipeable` (~3KB gzipped). Handles touch/pointer event normalization
across Android browsers. The existing hand-rolled `PullToRefresh.tsx` stays
as-is — different interaction pattern.

### Interaction model

- **Swipe right** → ACK (green background revealed). Active incidents only.
- **Swipe left** → UnACK (red background revealed). Acknowledged incidents only.
- Directional guard: wrong-direction swipe is ignored (e.g. swiping left on
  an active incident does nothing).
- Threshold-based: swipe past ~80px threshold triggers the action immediately.
  No confirmation dialog.
- **Tap still navigates** to detail screen. Swipe and tap don't conflict —
  `react-swipeable` distinguishes between the two.

### ACK parameters

| Parameter | Value | Source |
|---|---|---|
| `password` | API key | `settingsStorage` (localStorage) |
| `pin` | PIN (optional) | `settingsStorage` (localStorage) |
| `incident_id` | Incident ID | Incident row data |
| `user` | `"Thomas"` | Hardcoded (replaced by QR provisioning later) |
| `comment` | `""` | Empty (list swipe is one-tap, no comment) |

### After ACK/UnACK

1. Row shows brief spinner overlay
2. On success: invalidate React Query `['incidents']` cache → list refreshes
3. On failure: show toast/snackbar with error message, row resets to original
   position

### File changes

- New: `features/incidents/SwipeableIncidentRow.tsx` — wraps `IncidentRow`
  with swipe gesture handling and revealed action background.
- Modified: `IncidentListScreen.tsx` — use `SwipeableIncidentRow` instead of
  `IncidentRow` in the list.
- New: `lib/api/incidents.ts` — add `acknowledgeIncident()` and
  `unacknowledgeIncident()` functions.

## Feature 3: Polished Settings

### Fields

| Field | Type | Storage key | Notes |
|---|---|---|---|
| API Key | password input + eye toggle | `benem:bhnm-api-key` | Existing, enhanced with show/hide |
| PIN / License ID | text input | `benem:bhnm-pin` | New. Optional. Labeled "SaaS only" |

Both stored in localStorage. PIN is sent with all API calls when present
(same behavior as the current `VITE_BHNM_PIN` env var, but now user-editable).

### Test connection

- Button: "Test Connection" (full-width, below the credential fields)
- Calls `POST /api/proxy/ha-status` via middleware with current API key + PIN
- Three states:
  - **Testing**: spinner + "Testing connection..." text, button disabled
  - **Success**: green banner showing HA role and status from response
  - **Failed**: red banner with error type (auth failed, unreachable, server error)

### `ha_status` response parsing

Response: `[{"role": "master", "status": "1"}]` (array-wrapped).

Role display mapping:
- `standalone` → "Standalone"
- `primary` / `master` → "Primary"
- `slave` → "Replica"

Status display mapping (by role):
- Primary: `1` = "Active", `2` = "Inactive"
- Slave: `1` = "Active", `2` = "Takeover", `3` = "Inactive"
- Standalone: status not shown (always active by definition)

### Layout changes

- Grouped card layout (replaces flat form)
- "BHNM Connection" section header
- API key + PIN in a single card
- Test connection button below
- Save / Clear buttons
- "About" section at bottom: version number, platform label

### `config.ts` changes

`buildSnapshot()` must read PIN from localStorage (new `loadPin()` in
`settingsStorage.ts`) instead of only from `VITE_BHNM_PIN`. localStorage
value takes precedence over env var (same pattern as API key).

### File changes

- Modified: `features/settings/SettingsScreen.tsx` — new layout, PIN field,
  test-connection, about section
- Modified: `features/settings/settingsStorage.ts` — add `loadPin()`,
  `savePin()`, `clearPin()`
- Modified: `lib/config.ts` — read PIN from localStorage via `loadPin()`
- New: `lib/api/ha-status.ts` — `testConnection()` function

## Feature 4: iOS test-connection fix

Switch `ServerConfigView.swift` test-connection from `POST /restful/devices/list`
to `POST /api/ha_status_api.php`. Same auth flow, much lighter response.

This is a small, isolated change. The test-connection logic in
`ServerConfigView.swift` (lines ~254–388) builds a POST request to
`/fw/index.php?r=restful/devices/list` — change the path and adjust response
validation to expect `[{"role": "...", "status": "..."}]` instead of a device
list.

**Auth note:** `ha_status` uses `password=` (not `pwd=`), which is the same
param name the current `devices/list` call already uses. No auth change needed.

## Dependency: `react-swipeable`

```
npm install react-swipeable
```

~3KB gzipped. Well-maintained. Only new runtime dependency for v0.1.1.

## What's NOT in v0.1.1

- Alarms list and state log on detail screen (v0.1.2, needs `getincidentdetail`)
- Alarm count badges on detail screen (v0.1.2)
- QR code / `benem://` URL provisioning (future)
- Web Push notifications (separate feature track)
- ACK with comment (detail screen offers this via middleware, but list swipe is
  comment-free by design)
- Middleware caching (proxy routes are cache-ready but pass-through for now)
